# 一、整体级问题（贯穿 5 个页面）

这些是**全局性问题**，不解决会一直影响产品“专业度”。

---

## 1️⃣ 信息密度严重偏低（“网页感”过强）

**表现**

- 大量留白，但信息并不复杂
- 单个页面只承载极少操作（如日志页、历史版本页）
- 中央内容区域空旷，视觉重心漂浮

**问题本质**

- 这是典型的 **Web 表单布局**，不适合桌面工具类软件
- 桌面用户预期是：**效率优先、信息集中、少滚动**

**后果**

- 显得“像一个设置网页”，而不是“桌面工具”
- 高分辨率屏幕下显得“很空”“不专业”

**改进建议**

- 缩窄主内容最大宽度（如 960 / 1040）
- 使用 **分栏 / 表格 / 行内设置** 提高信息密度
- 桌面端允许“略密”，不是移动端

---

## 2️⃣ 左侧导航：选中态表达不清晰

**表现**

- 当前页仅用边框变化表示选中
- 颜色、背景、字体权重几乎一致

**问题本质**

- 桌面导航必须做到**一眼定位当前模块**
- 目前属于“弱状态区分”

**后果**

- 用户切页后需要“再确认自己在哪”
- 可用性不差，但**专业度低**

**改进建议**

- 选中项：

  - 左侧色条 / 背景色
  - 字重加粗

- 非选中项降低对比度

---

## 3️⃣ 缺乏「全局主操作」概念

**表现**

- 所有页面几乎都是“表单 + 按钮”
- 没有一个明显的“核心动作”

**问题本质**

- 工具类软件应始终有：

  - 当前页面**最重要的操作**
  - 明确的操作路径

**后果**

- 用户不知道“我现在应该干什么”
- 产品显得被动

**改进建议**

- 每页明确一个 Primary Action

  - 配置页：保存 & 测试
  - 历史版本：刷新 / 下载
  - 日志：打开 / 清空

# 二、页面逐一分析

---

## 页面 1：**AI 服务配置**

### 🔴 问题 1：表单缺乏语义分组层级

**表现**

- API 配置、提示词设置几乎同级
- 标题层级不明显

**问题**

- 对新用户来说理解成本高
- 不知道“哪些是必填，哪些是高级”

**改进**

- 明确分级：

  - 基础配置（Base URL / Key / Model）
  - 高级配置（System Prompt）

---

### 🔴 问题 2：API Key 的「眼睛图标」语义不够明确

**表现**

- 只有图标，没有文字提示
- 不确定是“查看 / 编辑 / 复制”

**问题**

- 桌面软件用户不一定熟悉 Web 交互隐喻

**改进**

- Hover 提示
- 或使用“显示 / 隐藏”文字

---

### 🟡 问题 3：底部按钮位置不符合操作节奏

**表现**

- 「测试连接」「保存配置」在右下角
- 与表单内容割裂

**问题**

- 用户填写时视线在中部，提交按钮在下方角落

**改进**

- 按钮紧贴表单区
- 主按钮视觉更强（颜色 / 实心）

---

## 页面 2：**系统设置**

### 🔴 问题 1：功能模块的“权重错位”

**表现**

- 右键菜单集成、外观设置、系统托盘并列
- 但重要性完全不同

**问题**

- 视觉上看不出“推荐 / 常用 / 高级”

**改进**

- 常用设置前置
- 高级/系统级选项弱化或折叠

---

### 🟡 问题 2：主题选择的即时反馈缺失

**表现**

- 选择主题后无明显变化反馈（至少截图中不可感知）

**问题**

- 用户会怀疑是否生效

**改进**

- 即时生效
- 或提示“需要重启”

---

## 页面 3 + 4：**历史版本（含横向扩展）**

### 🔴 问题 1：空状态设计过于“负面”

**表现**

- 大面积粉色背景 + 警告图标
- 文案是“暂无历史版本”

**问题**

- 这不是错误，只是“暂无数据”
- 使用错误态视觉会制造焦虑

**改进**

- 使用中性 Empty State
- 图标用灰色信息态，而非红色警告

---

### 🔴 问题 2：横向扩展后仍未利用空间

**表现**

- 窗口变宽，但内容仍居中一小块

**问题**

- 桌面软件必须 **响应窗口尺寸**
- 否则显得“假桌面应用”

**改进**

- 宽屏下：

  - 列表多列
  - 版本信息卡片化

---

### 🟡 问题 3：「刷新列表」按钮位置割裂

**表现**

- 按钮在右下角
- 与空状态内容不关联

**改进**

- 按钮作为 Empty State 的一部分
- 居中放置更合理

## 页面 5：**日志**

### 🔴 问题 1：日志级别选择逻辑不直观

**表现**

- Error / Warning 并列

**问题**

- 专业用户会困惑过滤逻辑

**改进**

- 明确说明：

  - 勾选表示包含

**表现**

- 没有日志预览区域

**问题**

- 桌面工具用户期望直接查看

- 内嵌日志查看区
- 文件操作作为补充

## 页面 6：**关于**

### 🟡 问题 1：信息呈现偏“网页 About”

**表现**

**问题**

- 紧凑信息结构

# 改进方案

1. 左侧导航选中态更明显（色条 + 字重）
   问题

当前选中态仅靠边框/弱对比，桌面“导航定位感”不足。

WPF-UI 最小落地

给 NavigationViewItem 的 IsActive/IsSelected 状态加：

左侧 3px 颜色条（Border）

字重 SemiBold

做法 A（推荐，纯样式）：在 NavigationView 的 ItemContainerStyle / Resources 里改 NavigationViewItem 模板（只加一个 LeftIndicator 的 Border）。
做法 B（更省）：如果你暂时不想动 ControlTemplate：

选中态背景 + 字重 + 前景色拉开对比（至少能救 80%）。

关键是：选中态必须“一眼看出”。

2. 每页只明确一个 Primary Action（实心主按钮）
   问题

“测试连接 / 保存配置”“刷新列表”等按钮同权，缺乏主动作引导。

WPF-UI 最小落地

用 WPF-UI 的按钮外观属性（通常是 Appearance / ButtonAppearance / Style，不同版本命名略有差异）：

AI 配置页：保存配置 = Primary（实心），测试连接 = Secondary/Outline

历史版本页：刷新列表 = Primary

日志页：打开日志文件 = Primary，清除日志 = Danger/Secondary（看你语义）

额外小技巧（不改布局也能提升专业度）

主按钮放右侧（符合 Windows 桌面习惯）

次按钮放左侧

3. 右侧内容区域加最大宽度（max-width 思路的 WPF 等价实现）
   问题

窗口拉宽后内容被拉得很散，视觉很“网页”。

WPF 最小落地

给右侧内容的根容器（比如 Grid / StackPanel 外层）加一层：

外层：Grid HorizontalAlignment="Stretch"

内层：Grid/StackPanel MaxWidth="1040" HorizontalAlignment="Center"

示例结构（概念）：

ContentHost（Stretch）

ContentContainer（Center + MaxWidth）

这样无需改变任何页面控件，只是“收口”。

4. 历史版本空状态：从“错误态”改为“中性空状态”
   问题

粉色大背景 + 警告 icon，表达成了“出错”，但其实是“暂无数据”。

WPF-UI 最小落地

不改布局，只改三样资源：

背景：用 Surface/Neutral（浅灰/卡片底），不要红/粉错误底

Icon：用 Info/Package 灰色，别用 Error/Warning 红色

文案：

「暂无历史版本」→「尚未获取历史版本」

副文案：「点击刷新获取列表」

把“刷新按钮”放在空状态文案下方居中（见改点 5）

空状态的语义正确 = 专业感直接拉满。

5. 按钮靠近它影响的内容（减少“漂在右下角”）
   问题

按钮固定在右下角，和内容断裂，操作路径长。

WPF 最小落地（不重构）

AI 配置页：把按钮放到表单卡片底部（卡片 Footer 区）

历史版本页：把 刷新列表 放进空状态卡片内部（文案下方）

日志页：打开/清空 放在“日志记录/级别”区块底部，而不是远离分组

实现上通常只是把 Button 从外层 Grid 的角落挪到对应 GroupBox/Card 的底部 StackPanel 里。

最小改动优先级（建议按这个顺序做）

导航选中态（全局收益最大）

Primary Button（全局交互立刻专业）

内容 MaxWidth（一秒去“网页感”）

空状态语义（信任感提升）

按钮就近（可用性提升）
